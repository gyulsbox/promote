## 코딩 관례 및 변수 선언 규칙

## CI 워크플로우에서 사용하는 CLI 버전 고정 규칙

- 모든 CI 워크플로우 파일(`.github/workflows/*.yml` 등)에서 사용하는 CLI 버전을 `package.json`에 명시된 버전(`^0.0.20` 등)과 일치하도록 고정한다.
- CLI 버전이 `npm install` 또는 관련 명령어로 설치될 때 버전이 고정되도록 명시하거나, 별도 버전 고정 명령어를 사용한다.
- 버전이 고정되지 않은 경우, 잠재적 문제 또는 주요 변경사항으로 간주하여 수정한다.
- 버전 고정이 누락된 경우, PR 검토 시 반드시 지적하고 수정 요청한다.
- 이 규칙은 저장소 전체에 적용하며, 특정 PR에 한정하지 않는다.


## 테스트 리소스 관리 패턴 안내

- 모든 통합 테스트에서 서버와 클라이언트 리소스 관리를 위해 `testServerAndClientResource` 패턴을 일관되게 사용하세요.
- 개별 테스트에서 서버와 클라이언트의 생명주기를 수동으로 관리하는 코드를 제거하고, `testServerAndClientResource`를 활용하세요.
- `await using` 구문과 함께 `testServerAndClientResource`를 적용하여 리소스 해제와 정리를 자동화하세요.
- `konn()` 또는 유사한 커스텀 래퍼 대신 표준 패턴인 `testServerAndClientResource`를 사용하여 일관성을 유지하세요.
- 특정 경로나 PR에 국한되지 않으며, 저장소 전체 테스트 코드에 이 패턴을 적용하여 유지보수성과 코드 품질을 향상시키세요.


## Node.js 버전 및 타입 정의 일관성 규칙

- `@types/node` 버전은 반드시 프로젝트의 Node.js 주요 버전과 일치시켜야 하며, `package.json`의 `engines.node` 설정과 일치하도록 유지한다.
- `@types/node` 버전은 Node.js 버전이 24.x인 경우 `^24.x`로, 25.x인 경우 `^25.x`로 업데이트한다.
- `@types/node`와 `engines.node` 간 버전 불일치가 발견되면 즉시 수정하거나 관련 PR에 적절한 수정 내용을 반영한다.
- 이 규칙은 `packages/*` 또는 전체 프로젝트 내의 모든 관련 파일(`package.json`, `tsconfig.json` 등)에 적용하며, 특정 경로나 PR에 한정하지 않는다.
- 버전 일관성 유지에 관한 내용은 `AGENTS.md` 또는 관련 문서에 명확히 기록하여 개발자와 AI 에이전트가 참고할 수 있도록 한다.


## pnpm 버전 일치 규칙

- `.tool-versions` 파일 내 pnpm 버전과 `package.json`의 `engines.pnpm` 또는 `packageManager` 필드의 버전을 반드시 일치시킨다.
- `.tool-versions`에 pnpm 8.15.5로 명시되어 있다면, `package.json`의 `packageManager`와 `engines.pnpm`도 동일한 버전(예: pnpm@8.15.5 또는 ^8.15.5)으로 수정한다.
- 버전 불일치가 발견될 경우, 즉시 `.tool-versions` 또는 `package.json`을 수정하여 일치시킨다.
- 이 규칙은 레포지토리 전반에 적용하며, 새로 PR을 생성하거나 수정할 때 반드시 준수한다.
- 규칙 위반 시, AI 에이전트는 수정 요청 또는 알림을 통해 일치시킬 것을 권고한다.


## 패키지 배포 규칙: bin 실행 파일 포함

- `package.json`의 `files` 배열에 반드시 `bin` 디렉토리(`bin/`)를 포함시킨다.
- `bin/` 디렉토리 내의 실행 파일들이 배포 시 누락되지 않도록 한다.
- `bin`이 누락될 경우, `bin` 엔트리로 지정된 실행 파일이 배포되지 않아 바이너리 작동에 문제가 발생한다.
- 이 규칙은 여러 패키지에 공통적으로 적용되며, repo 전체에 일관성을 유지한다.
- `files` 배열에 `bin`을 추가하는 변경 사항을 반드시 커밋하여 배포 전에 반영한다.


## 바이너리 네이밍 규칙

- 여러 `@trpc/*` 패키지에서 동일한 바이너리 이름(예: `intent`)을 노출하지 않도록 하며, 고유한 네임스페이스가 포함된 이름을 사용한다. 예: `@trpc/intent-cli`, `@trpc/intent-tool`.
- 바이너리 이름은 `package.json`의 `bin` 필드에 명확히 정의하며, 충돌 방지를 위해 일관된 네이밍 규칙을 따른다.
- 전체 저장소에 걸쳐 일관된 네이밍 규칙을 적용하여, 여러 패키지 간 바이너리 충돌 가능성을 방지한다.
- 바이너리 이름이 충돌할 경우, 사용자 환경에서 `node_modules/.bin/` 내에서 충돌이 발생할 수 있음을 유념한다.
- 이 규칙은 특정 PR이나 경로에 국한되지 않고, 전체 저장소의 패키지에 적용한다.


## AGENTS.md 규칙

- `subscribe()` 호출 시 `onError` 핸들러 내에서 `subscription.unsubscribe()`를 호출할 때, `subscription` 변수를 `const` 대신 `let`으로 선언하도록 한다.
- `subscription` 변수는 여러 위치에서 재할당 가능하도록 `let`으로 선언하며, `subscribe()` 호출과 동시에 선언한다.
- 이 규칙은 여러 테스트 파일과 코드 전반에 일관성을 유지하기 위해 적용하며, 특정 경로나 PR에 국한하지 않는다.
- `subscription` 변수 선언과 `unsubscribe()` 호출이 같은 코드 블록 내에 위치하도록 하며, 명확한 구독 취소 시점을 확보한다.
- 이 규칙을 위반하는 코드는 수정하여 `subscription`을 `let`으로 선언하고, `unsubscribe()` 호출 시점에 재할당 가능하게 한다.


## 파일명 명명 규칙 예외 처리

- `SKILL.md` 파일명은 md/mdx camelCase 명명 규칙을 위반하더라도, 의도된 예외로 명확히 문서화한다.
- 이 예외는 여러 경로에 걸쳐 반복되며, 전사적 코딩 가이드라인에 따라 `AGENTS.md`와 같은 AI 에이전트 지침 문서에 포함한다.
- `SKILL.md` 파일명을 camelCase 규칙에 맞게 변경하지 않으며, 예외로 처리한다.
- `SKILL.md`를 규칙 예외로 명시할 경우, 관련 가이드라인에 반드시 추가한다.
- 이 규칙은 프로젝트 전반에 걸쳐 일관되게 적용하며, 예외 처리에 대한 명확한 문서화를 유지한다.
