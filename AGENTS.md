## pnpm 버전 일치 정책

- `.tool-versions` 파일 내 pnpm 버전과 `package.json`의 `packageManager` 또는 `engines.pnpm` 버전을 반드시 동일하게 유지하세요.
- `.tool-versions`에 명시된 pnpm 버전이 최신 또는 프로젝트 요구 사항에 맞게 업데이트되었는지 정기적으로 확인하세요.
- 버전 불일치가 발견되면 즉시 `.tool-versions` 또는 `package.json`을 수정하여 일치시켜 주세요.
- 이 정책은 레포지토리 전반에 걸쳐 일관된 도구 버전 관리를 위해 적용됩니다.
- 버전 불일치로 인한 재현성 문제를 방지하기 위해 PR 시 반드시 버전 일치 여부를 검증하세요.


## 패키지 배포 규칙: bin 디렉토리 포함 필수

- `package.json`의 `bin` 필드에 명시된 실행 파일 경로(`./bin/`)는 반드시 `files` 배열에 포함시켜야 한다.
- `files` 배열에 `bin/` 디렉토리가 누락되면, 배포 시 해당 실행 파일이 누락되어 바이너리 깨짐 문제가 발생한다.
- `bin/` 디렉토리를 `files`에 추가하는 것을 잊지 말고, 배포 전 반드시 검증한다.
- 이 규칙은 모든 패키지에 일관되게 적용되어야 하며, 배포 관행의 표준으로 준수한다.
- 위 규칙 위반 시, 배포된 패키지의 바이너리 파일이 누락되어 사용자에게 문제가 발생할 수 있다.


## 실행 파일 네이밍 규칙

- 여러 패키지에서 동일한 이름(`intent`)의 실행 파일을 노출하는 경우, `node_modules/.bin` 내에서 충돌이 발생할 수 있으므로, 고유한 네임스페이스가 포함된 이름을 사용하세요.
- 예를 들어, `@trpc/` 관련 패키지에서는 `@trpc/intent`와 같이 네임스페이스를 명확히 구분하는 이름을 권장합니다.
- 충돌 방지를 위해, 패키지별로 고유한 실행 파일 이름을 정하고, `package.json`의 `bin` 필드에 명시하세요.
- 동일한 프로젝트 내에서 여러 패키지가 `intent`라는 이름의 실행 파일을 노출하는 것을 피하세요.
- 이 규칙은 저장소 전반에 걸쳐 일관되게 적용되어, 향후 개발자와 AI 에이전트가 참고할 수 있도록 문서에 명시하세요.


## 구독 취소 처리 방식 표준화

- `subscribe()` 호출 시 `onError` 핸들러 내에서 `subscription.unsubscribe()`를 호출할 때, `subscription` 변수를 `const` 대신 `let`으로 선언한다.
- 전역적으로 반복되는 구독 취소 패턴에 대해 일관된 코딩 관례를 유지한다.
- 특정 파일이나 PR에 국한하지 않고, 전체 저장소에 적용할 수 있도록 지침을 준수한다.
- `subscribe()` 호출과 `subscription` 선언을 별도로 분리하지 않고, `let` 선언 후 바로 `subscribe()`를 호출하는 방식을 권장한다.
- 구독 변수 선언 시, `const` 대신 `let`을 사용하여 `onError` 핸들러 내에서 안전하게 `unsubscribe()` 호출이 가능하도록 한다.


## maxBatchSize 검증 규칙

- `maxBatchSize`가 유한한 양의 정수인지 여부를 반드시 검증한다.
- `typeof maxBatchSize === 'number'`만으로는 `NaN`, `Infinity`, 음수, 0, 비정수 값을 걸러내지 않으므로, 추가로 `Number.isInteger(maxBatchSize)`와 `maxBatchSize > 0` 검증을 수행한다.
- 검증은 `/path/to/your/file.js` 또는 관련 파일 내에서 일관되게 적용한다.
- 검증 로직은 `maxBatchSize` 사용 전에 수행하며, 유효하지 않은 경우 예외를 던지거나 적절한 기본값으로 대체한다.
- 이 규칙은 여러 기능과 파일에 반복 적용되어 코드 품질과 안정성을 확보한다.


## 파일명 규칙 및 예외 처리

- `SKILL.md`와 같은 파일명은 camelCase 규칙을 위반하므로, 해당 파일명을 camelCase로 변경하거나, 규칙 예외 목록에 명시적으로 추가해야 한다.
- 이 규칙은 전체 저장소에 적용되며, 여러 PR에서 반복적으로 나타나는 문제이므로 반드시 준수할 것.
- `SKILL.md`와 유사한 파일명에 대해서도 동일하게 적용하며, 예외 처리 시 명확한 문서화가 필요하다.
- 규칙 위반 시, 파일명을 적절히 수정하거나, 예외 목록에 추가하는 작업을 수행할 것.
- 관련 규칙 및 예외 처리 내용은 `AGENTS.md` 또는 저장소 전체 규칙 문서에 포함시켜 관리할 것.


## 의존성 변경 시 패키지 파일 동기화 규칙

- 의존성을 변경한 경우, 반드시 `package.json`과 `pnpm-lock.yaml` 파일을 동시에 업데이트하고 커밋하세요.
- 의존성 변경 후 `pnpm install` 또는 `pnpm update` 명령어를 실행하여 lockfile을 재생성하세요.
- 변경된 `pnpm-lock.yaml` 파일이 `package.json`과 일치하는지 반드시 확인하세요.
- CI 실패 방지를 위해, PR 제출 전에 항상 lockfile이 최신 상태인지 검증하세요.
- 이 규칙은 전체 저장소에 적용되며, 개별 PR 또는 특정 경로에 한정되지 않습니다.


## 릴리스 관련 워크플로우에서 npm 버전 관리 지침

- `pnpm add -g npm@latest` 명령은 사용하지 않으며, 릴리스의 재현 가능성을 저해하고 예기치 않게 깨질 수 있으므로 피할 것.
- 안정적인 npm 버전은 특정 버전으로 고정하거나, Node.js에 번들된 npm 버전을 사용할 것.
- 전체 저장소의 릴리스 및 배포 관련 워크플로우 파일에서 일관되게 적용할 것.
- 버전 고정은 `package.json` 또는 관련 스크립트 내에서 명확히 지정할 것.
- 릴리스의 결정론적이고 안정적인 버전 관리를 위해 이 지침을 준수할 것.


## CI 워크플로우에서 사용하는 CLI 버전 고정 규칙

- `/.github/workflows/` 내 모든 워크플로우 파일에서 CLI 버전이 `package.json`에 명시된 버전(`^0.0.20` 등)과 일치하도록 고정한다.
- `npm install` 또는 `npx` 명령어를 사용할 때, 버전 번호를 명시하여 최신 버전이 아닌 특정 버전이 설치되도록 한다.
- 버전 미고정 또는 최신 버전 설치가 발견될 경우 즉시 수정하여, 버전 고정을 강제한다.
- CLI 관련 명령어 또는 스크립트는 `/.github/scripts/` 또는 워크플로우 파일 내 명시된 위치에서 버전이 일치하는지 검증한다.
- 이 규칙을 위반하는 경우, 수정 요청 또는 자동 수정 스크립트 실행을 통해 버전 일치를 유지한다.


## 마크다운 코드 블록 언어 식별자 규칙

- 모든 마크다운 문서 내 코드 블록에는 적절한 언어 식별자를 추가하세요. 예: ```python, ```javascript, ```bash 등.
- `docs/`, `src/`, `README.md` 등 특정 파일 또는 폴더에 상관없이 일관되게 적용하세요.
- 코드 블록이 언어 식별자를 누락된 경우, markdownlint MD040 경고가 발생하며 편집기 지원이 저하됩니다.
- 새 문서 또는 기존 문서 수정 시, 코드 블록 시작 부분에 언어 식별자를 반드시 포함하세요.
- 규칙 위반 시 경고 메시지(예: "Add a language hint to the fenced code block")를 참고하여 수정하세요.
