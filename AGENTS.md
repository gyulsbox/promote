## 도구 체인 버전 일치 규칙

- `package.json`의 `engines.packageManager` 또는 `engines.pnpm`과 `.tool-versions` 파일 내 `pnpm` 버전이 일치하도록 유지한다.
- 두 파일 간 버전 불일치가 발견되면 즉시 수정하여 일관성을 확보한다.
- 버전 변경 시, 변경 사항을 커밋 메시지에 명확히 기록한다.
- 이 규칙은 레포지토리 전체에 적용되며, 특정 폴더 또는 PR에 한정하지 않는다.
- 도구 체인 관리에 관한 전사적 규칙으로, 모든 개발자가 준수하도록 한다.


## Node.js 런타임 버전과 @types/node 타입 정의 버전 일치 규칙

- 프로젝트의 `package.json` 파일 내 `engines.node` 버전과 `@types/node` 버전이 일치하거나 호환되는 버전으로 유지하세요.
- Node.js 런타임 버전이 `^24.0.0`인 경우, `@types/node`도 `^24.x` 버전으로 업데이트하세요.
- 버전 불일치로 인한 빌드 또는 타입 오류를 방지하기 위해 정기적으로 버전 검증 및 업데이트를 수행하세요.
- `@types/node` 버전은 Node.js 런타임 버전의 주요 버전과 일치해야 하며, 예를 들어 Node.js 25를 사용하는 경우 `@types/node`도 `^25.x`로 설정하세요.
- 이 규칙을 준수하지 않으면 코드 품질 저하와 빌드 실패 가능성이 높아지므로 반드시 지침에 따라 관리하세요.


## 배포 시 `bin` 디렉터리 포함 규칙

- `bin` 디렉터리에 있는 실행 파일을 배포 패키지에 포함시키기 위해, 반드시 `package.json`의 `files` 배열에 `bin` 또는 `bin/` 경로를 추가한다.
- `files` 허용 목록에 `bin/`이 포함되지 않으면, 배포 시 실행 파일이 누락되어 소비자에게 문제가 발생하므로 반드시 포함시킨다.
- `bin` 디렉터리 내의 파일이 `package.json`의 `bin` 필드와 일치하는지 확인한다.
- `files` 배열에 `bin/`을 추가하는 방법은 예를 들어 `"files": ["lib", "bin"]`과 같이 작성한다.
- 이 규칙은 여러 패키지에 일관되게 적용하여 배포 품질을 유지하는 중요한 관행임을 유념한다.


## 바이너리 네이밍 규칙

- 여러 `@trpc/*` 패키지에서 동일한 `intent` 바이너리를 노출하지 않도록 하며, 네임스페이스가 포함된 고유한 이름을 사용하세요 (예: `@trpc/intent-cli`).
- 바이너리 이름은 패키지별로 고유하게 지정하여 `node_modules/.bin/` 내 충돌을 방지하세요.
- 바이너리 이름이 충돌 가능성이 있는 경우, 관련 패키지 전체에 일관된 네이밍 규칙을 적용하세요.
- 바이너리 이름을 변경할 때는 사용자 환경에서의 충돌 방지와 명확한 구분을 위해 명확한 네임스페이스를 포함하세요.
- 충돌 방지와 명확성을 위해, 모든 `@trpc/*` 패키지의 바이너리 이름은 고유하고 일관된 네이밍 규칙을 따르도록 문서화하세요.


## AGENTS.md: 패턴 일관성 및 bin 설정 가이드라인

- 여러 `@trpc/*` 패키지에서 동일한 `bin` 이름(예: `intent`)을 중복 선언하지 않도록 한다.
- `bin`으로 선언된 파일(`./bin/intent.js`)은 반드시 해당 경로에 존재하도록 하며, 누락되지 않도록 한다.
- `bin` 선언 시, 선언된 파일이 PR에 포함되어 있는지 반드시 확인한다.
- 동일한 `bin` 이름을 사용하는 경우, 충돌 방지 및 명확한 역할 분담을 위해 별도 가이드라인을 따른다.
- `bin` 관련 문제 발생 시, 즉시 수정하여 배포 전 검증을 완료한다.


## 구독 변수 선언 및 에러 핸들러 처리 규칙

- `subscribe()` 호출과 동시에 `const`로 선언된 `subscription` 변수를 `onError` 핸들러 내에서 참조하지 않도록 한다.
- `subscription` 변수는 `let`으로 선언하거나, 별도 초기화 후 참조하도록 수정한다.
- 구독 취소(`unsubscribe()`)는 `onError` 핸들러 내에서 `subscription`이 안전하게 참조된 후 수행한다.
- 이 규칙은 모든 테스트 파일에서 일관되게 적용하며, 전역적인 코드 안전성과 가독성을 위해 준수한다.
- `subscribe()` 호출과 `subscription` 선언, `unsubscribe()` 호출의 순서와 위치를 명확히 하여, 타이밍 문제를 방지한다.


## AI 에이전트 지침 규칙

- `SKILL.md` 파일명은 md/mdx camelCase 명명 규칙을 위반하지만, 이는 의도된 예외이므로 `AGENTS.md`에 명확히 문서화한다.
- `SKILL.md`와 유사한 예외 파일명은 모두 `AGENTS.md`에 기록하여 일관성을 유지한다.
- 파일명 규칙 위반 시, 무조건 이름을 변경하지 말고, 예외로 처리할 경우 해당 내용을 명확히 문서화한다.
- `AGENTS.md`에 명시된 예외 규칙은 전사적 코딩 가이드라인에 반영하여 모든 팀원이 참고할 수 있도록 한다.
- 예외 처리 대상 파일은 패턴 또는 특정 파일명(`SKILL.md`)으로 한정하며, 다른 파일명은 규칙 준수 여부를 검토한다.


## 의존성 변경 시 파일 동기화 규칙

- 의존성을 변경할 때는 반드시 `package.json`과 `pnpm-lock.yaml` 파일을 동시에 업데이트하고 커밋하세요.
- 의존성 변경 후 `pnpm install` 또는 `pnpm update` 명령어를 실행하여 lockfile을 재생성하세요.
- 변경된 `pnpm-lock.yaml` 파일이 `package.json`과 일치하는지 반드시 확인하세요.
- CI 실패 방지를 위해, PR 제출 전에 `pnpm-lock.yaml`이 최신 상태인지 검증하세요.
- 모든 패키지 관리 작업은 프로젝트 루트 또는 관련 패키지 디렉터리에서 수행하며, 변경 사항을 잊지 말고 커밋하세요.


## CI 워크플로우에서 사용하는 CLI 버전 고정 규칙

- 모든 CI 워크플로우 파일(예: `.github/workflows/*`)에서 사용하는 CLI 버전을 `package.json`에 명시된 버전(`^0.0.20` 등)과 일치하도록 고정하세요.
- `npm install` 또는 관련 명령어를 사용할 때, CLI 패키지의 버전이 최신이거나 미지정 상태로 설치되지 않도록 주의하세요.
- 버전이 일치하지 않는 경우, CI 스크립트 내에서 명시적으로 버전 번호를 지정하거나, `package.json`의 버전과 일치하는지 검증하는 절차를 포함하세요.
- 버전 고정이 누락된 경우, PR 검토 시 반드시 지적하고 수정 요청을 하세요.
- 이 규칙은 저장소 전체에 적용되며, 특정 PR에 한정하지 않고 일관된 버전 관리를 유지하세요.


## 마크다운 코드 블록 언어 식별자 규칙

- 모든 마크다운 코드 블록(예: ` ``` `)에 적절한 언어 식별자(예: ` ```python `, ` ```javascript ` 등)를 명시하세요.
- `markdownlint MD040` 경고를 방지하기 위해, 코드 블록이 있는 모든 문서에서 언어 식별자를 빠짐없이 추가하세요.
- `docs/`, `src/`, `README.md` 등 주요 문서 파일 내의 코드 블록에 일관되게 적용하세요.
- 언어 식별자를 생략하거나 잘못된 언어를 지정하는 경우 수정하여 경고를 제거하세요.
- 에디터 지원과 가독성을 높이기 위해, 항상 명확한 언어 식별자를 사용하세요.
